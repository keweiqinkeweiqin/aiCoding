## 一、系统概述

华尔街之眼是一款 AI 驱动的投研情报引擎，面向科技/AI 赛道投资者，提供多源新闻聚合、事件聚类、置信度评估、个性化研判等能力。

核心价值链：**多源采集 → 四级去重 → LLM 结构化提取 → Embedding 向量化 → 事件聚类 → 置信度评估 → 个性化排序 → AI 研判**

#### 5.2.2 四级去重算法 (DeduplicationService)

```
输入: title, url, content
  │
  ├─ Level 1: URL 精确匹配
  │   newsArticleRepository.existsBySourceUrl(url)
  │   → 命中返回 "URL_MATCH"
  │
  ├─ Level 2: SimHash 标题指纹
  │   simHash(title) → 64位指纹
  │   遍历内存缓存，hammingDistance ≤ 2 → "TITLE_SIMHASH"
  │   注：阈值收紧到2，给交叉验证留空间（交叉验证用≤10）
  │
  ├─ Level 3: MD5 内容哈希
  │   MD5(normalize(content)) → 32位hex
  │   newsArticleRepository.existsByContentHash(hash)
  │   → 命中返回 "CONTENT_HASH"
  │
  └─ Level 4: SimHash 内容指纹
      simHash(content) → 64位指纹
      遍历内存缓存，hammingDistance ≤ 3 → "CONTENT_SIMHASH"

SimHash 算法：
  1. 中文 bigram 分词（相邻两字符）
  2. 每个 token 做 MurmurHash64 → 64位值
  3. 加权合并：对应位出现+1，否则-1
  4. 最终每位>0取1，否则取0 → 64位指纹
```

#### 5.2.3 四维置信度评估 (CredibilityService)

| 维度 | 权重 | 算法 |
|------|------|------|
| 来源权威性 | 30% | 预设分数表：财联社0.95、东方财富研报0.92、华尔街见闻0.85、金十0.82、36Kr0.80、NewsAPI0.75，默认0.6 |
| LLM 情感置信度 | 25% | `0.5 + |sentimentScore - 0.5|`，越极端说明 LLM 越确定，范围 0.5-1.0 |
| 时效性 | 20% | 指数衰减：≤1h→1.0, ≤6h→0.9, ≤24h→0.7, ≤72h→0.5, >72h→0.3 |
| 交叉验证 | 25% | 48h内不同来源报道同一事件（SimHash汉明距离≤10）：≥3源→1.0, ≥2源→0.85, ≥1源→0.7, 单源→0.4 |

综合分数 = Σ(维度分 × 权重)，等级：≥0.8 authoritative / ≥0.5 normal / <0.5 questionable

#### 5.2.4 事件聚类算法 (EventClusterService)

```
输入: 最近48h未聚类的新闻
目标: 将同一事件的多条新闻聚合为一条情报

for each 未聚类新闻:
  │
  ├─ Primary: Embedding 余弦相似度匹配
  │   计算新闻向量 vs 所有已有情报向量
  │   最高相似度 ≥ 0.85 → 合并到该情报
  │
  ├─ Fallback: SimHash 汉明距离匹配
  │   simHash(title) vs 情报的 titleSimhash
  │   汉明距离 ≤ 20 → 合并到该情报
  │
  └─ 都不匹配 → 创建新情报

创建新情报时:
  - 用结构化文本（标题+摘要+标签+股票）生成情报向量
  - 同步写入 VectorSearchService 全局缓存（搜索/问答可立即命中）
  - 同步写入 EventClusterService 本地缓存（聚类匹配用）

合并新文章时:
  - 更新 sourceCount、credibilityScore（取最高）
  - 情感多数投票
  - 合并 tags 和 relatedStocks（去重）
  - 多来源(≥2)时 LLM 生成聚合标题+摘要
  - 清空 content 缓存（下次访问时重新生成）
  - 用更新后的结构化字段重新生成情报向量，替换旧向量
    （确保向量始终反映情报的最新语义，而非首篇文章的旧标题）

聚类完成后:
  - 异步并发预生成情报正文（线程池 size=3）
  - 多来源: LLM 综合生成投研分析（400-600字）
  - 单来源: LLM 改写为投研快讯（200-300字）
  - LLM 失败: 拼接原始新闻内容
```

#### 5.2.5 智能问答流程 (SmartQueryService)

```
用户提问
  │
  ▼
混合检索 (hybridSearch)
  ├─ 语义搜索: Embedding(question) → 直接搜情报向量 Top30
  │   → 返回 intelligenceId + 相似度分数
  │   → 过滤 similarity < 0.3
  │
  └─ 关键词 fallback: JPQL LIKE 搜索情报标题/摘要/标签/股票
      → 基础相关性分数 0.35
  │
  ▼
合并去重，按相关度排序，取 Top10
  │
  ▼
构建 LLM 上下文（情报级别，信息密度远高于原始新闻）
  │
  ▼
流式调用 LLM (stream=true)
  ├─ SSE event: meta      → 检索元数据（matchedCount + relatedNews）
  ├─ SSE event: reasoning → 模型思考过程（Qwen/DeepSeek 支持）
  ├─ SSE event: token     → 逐 token 输出回答
  └─ SSE event: done      → 完成

多轮对话: POST /api/chat/stream
  - history 数组传入上下文（最多保留最近5轮=10条消息）
  - 旧 assistant 消息截断到500字节省 token
```

#### 5.2.6 个性化排序算法 (IntelligenceController)

```
输入: 情报列表 + 用户画像(focusAreas, holdings)

for each 情报:
  score = 0
  + 标签匹配用户关注领域: +10分/个
  + 关联股票匹配用户持仓: +20分/个（持仓权重更高）
  + 高优先级(high): +5分
  + 多来源(sourceCount>1): +3分

按 score 降序排列，同分按时间倒序

C端首页(scene=home): 排序后取前3条
控制面板(scene=admin): 排序后全量分页
```

#### 5.2.7 个性化研判流程 (AnalysisService)

```
GET /api/intelligences/{id}/analysis?userId=1
  │
  ├─ 24h 缓存检查 (AnalysisRecord)
  │   命中 → 直接返回
  │
  ├─ 加载用户画像 + 持仓
  │
  ├─ 构建 Prompt:
  │   System: "AI投研助手，严格返回JSON"
  │   User: 情报标题+摘要+股票+情感 + 画像类型/周期/关注领域 + 持仓明细(代码+占比)
  │
  ├─ 调用 LLM → 解析 JSON
  │   {analysis, impacts[{stock,impact,level,volatility}], suggestion, risks[], userContext}
  │
  └─ 持久化到 AnalysisRecord
```

### 5.3 向量搜索设计 (VectorSearchService)

双缓存架构，分别存储新闻向量和情报向量：

```
新闻向量缓存: ConcurrentHashMap<Long(articleId), float[]>
  - 启动时从 DB 加载所有 NewsArticle.embeddingJson (@PostConstruct)
  - 新文章入库时同步写入缓存
  - 用途: 语义去重

情报向量缓存: ConcurrentHashMap<Long(intelligenceId), float[]>
  - 启动时从 DB 加载所有 Intelligence.embeddingJson (@PostConstruct)
  - 聚类创建/更新情报时由 EventClusterService 同步写入
  - 用途: 搜索、智能问答、事件聚类匹配

搜索方法:
  - semanticSearchWithScores(queryText, topK) → 搜新闻向量（去重用）
  - searchIntelligences(queryText, topK) → 搜情报向量（搜索/问答用）
  - findSemanticDuplicate(text, threshold) → 语义去重

余弦相似度: dot(a,b) / (||a|| × ||b||)
```

情报向量生成策略：
- 输入文本 = 标题 + 摘要 + 标签 + 关联股票（结构化拼接，截断到 500 字）
- 创建新情报时：用首篇文章的结构化字段生成
- 合并新文章后：用更新后的聚合标题/摘要/标签重新生成，替换旧向量
- 比纯标题向量语义覆盖面更广，聚类和搜索匹配更准确

说明：当前为内存暴力搜索，适合万级数据量。若数据量增长到十万级以上，建议引入 HNSW 索引或外部向量数据库（如 Milvus/Qdrant）。

## 十、已知限制与演进方向

### 10.1 当前限制

| 限制 | 说明 |
|------|------|
| 向量搜索 | 内存暴力遍历，适合万级数据量 |
| 数据库 | H2 单文件，不支持并发写入，不适合生产环境 |
| 认证 | 无密码、无 Token，仅 userId query param |
| 去重缓存 | SimHash 指纹存内存，重启后丢失（DB 中有 contentHash 兜底） |
| LLM 调用 | 无限流控，高并发时可能触发 API 限流 |
| 前端 | 原生 HTML/JS，无组件化，维护成本随页面增多而上升 |

### 10.2 演进方向

| 方向 | 方案 |
|------|------|
| 向量数据库 | 引入 Milvus/Qdrant，支持十万级以上向量检索 |
| 生产数据库 | 迁移到 PostgreSQL/MySQL |
| 认证鉴权 | JWT Token + Spring Security |
| 前端框架 | 迁移到 React/Vue SPA，组件化开发 |
| 缓存层 | Redis 缓存热点情报、研判结果 |
| 消息队列 | RabbitMQ/Kafka 解耦采集与处理 |
| 监控告警 | Prometheus + Grafana |
| 容器化 | Docker + K8s 部署 |
