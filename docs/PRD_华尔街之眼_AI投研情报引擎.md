## 一、产品概述

### 1.1 产品定位
面向 AI/科技领域个人投资者和研究人员的一站式投研情报引擎。通过多源信息采集、智能质量把控、用户画像驱动的个性化研判，帮助用户快速获取高质量投资情报并辅助决策。

### 1.2 核心价值主张
- **信息聚合**：多源采集 AI/科技投资领域资讯，一处看全
- **质量可信**：每条信息标注来源、置信度、去重处理，拒绝噪音
- **个性研判**：结合用户持仓和偏好，生成差异化的分析和操作建议
- **趋势可视**：关键指标图表化，辅助判断市场走向

### 1.3 目标用户
| 用户类型 | 特征 | 核心诉求 |
|---------|------|---------|
| 科技股个人投资者 | 持有 NVIDIA/TSMC/MSFT 等科技股 | 快速获取影响持仓的关键信息 |
| AI 行业研究员 | 关注 AI 产业链动态 | 多源交叉验证，深度研判 |
| 量化/趋势交易者 | 关注市场情绪和技术面 | 情绪指标、趋势图表辅助决策 |

## 三、核心功能详细设计

### 3.1 多源信息采集（必须项 ✅）

**要求：至少接入 2 个不同类型信息源**

| 数据源 | 类型 | 采集内容 | 频率 |
|-------|------|---------|------|
| 财经新闻 API（如 NewsAPI / Alpha Vantage News） | 新闻资讯 | AI/科技公司相关新闻标题、摘要、链接 | 每30分钟 |
| 社交媒体/论坛（如 Reddit r/wallstreetbets API / Twitter API） | 社区舆情 | AI/科技股讨论热度、观点倾向 | 每1小时 |
| 股票行情 API（如 Alpha Vantage / Yahoo Finance） | 市场数据 | AI/科技股实时/延迟行情、涨跌幅 | 每5分钟 |

**数据模型 — IntelItem（情报条目）：**
```
{
  id: Long,
  title: String,              // 标题
  summary: String,            // 摘要
  content: String,            // 正文/详情
  sourceType: Enum,           // NEWS / SOCIAL / MARKET_DATA / RESEARCH
  sourceName: String,         // 具体来源名称（如 "Reuters", "Reddit"）
  sourceUrl: String,          // 原始链接（信息溯源）
  credibility: Enum,          // AUTHORITATIVE / NORMAL / DOUBTFUL（权威/一般/存疑）
  confidenceScore: Double,    // 置信度 0.0~1.0
  relatedStocks: List<String>,// 关联股票代码
  tags: List<String>,         // 标签（AI, 芯片, 云计算...）
  publishedAt: DateTime,      // 原始发布时间
  collectedAt: DateTime,      // 采集时间
  contentHash: String,        // 内容哈希（用于去重）
  verified: Boolean           // 是否经过交叉验证
}
```

### 3.2 内容质量把控（必须项 ✅）

**要求：去重 + 信息溯源 + 置信度标注**

#### 3.2.1 去重机制
- 基于 `contentHash`（标题 + 摘要的 MD5/SimHash）进行精确去重
- 基于文本相似度（余弦相似度 > 0.85）进行近似去重
- 同一事件多源报道合并为一条，保留所有来源链接

#### 3.2.2 信息溯源
- 每条情报必须标注 `sourceUrl`（原始出处链接）
- 前端展示时显示来源名称，点击可跳转原文
- 多源报道同一事件时，列出所有来源

#### 3.2.3 置信度标注
LLM 辅助评估置信度，评估维度：
| 维度 | 权重 | 说明 |
|------|------|------|
| 来源权威性 | 40% | 权威媒体 > 一般媒体 > 自媒体/论坛 |
| 信息一致性 | 30% | 多源交叉验证一致则加分 |
| 时效性 | 20% | 越新越可信（衰减函数） |
| 内容完整性 | 10% | 有数据支撑 > 纯观点 |

**来源可信度分级规则：**
| 等级 | 标签 | 典型来源 |
|------|------|---------|
| 权威 (AUTHORITATIVE) | 🟢 | Reuters, Bloomberg, 官方财报, SEC Filing |
| 一般 (NORMAL) | 🟡 | 科技媒体, 行业分析师报告, 知名博主 |
| 存疑 (DOUBTFUL) | 🔴 | 匿名论坛, 未经证实的小道消息 |

### 3.3 用户画像与个性化研判（必须项 ✅）

**要求：支持用户画像输入，生成差异化研判和操作建议**

#### 3.3.1 用户画像模型 — UserProfile
```
{
  id: Long,
  username: String,
  focusAreas: List<String>,       // 关注领域：["AI芯片", "云计算", "自动驾驶"]
  holdings: List<Holding>,        // 持仓信息
  investmentStyle: Enum,          // CONSERVATIVE / BALANCED / AGGRESSIVE
  riskTolerance: Enum,            // LOW / MEDIUM / HIGH
  investmentHorizon: Enum,        // SHORT_TERM / MEDIUM_TERM / LONG_TERM
  preferredAnalysisType: Enum     // FUNDAMENTAL / TECHNICAL / MIXED
}

Holding {
  stockCode: String,    // 如 "NVDA"
  stockName: String,    // 如 "NVIDIA"
  quantity: Integer,     // 持仓数量
  avgCost: Double,       // 持仓成本
  sector: String         // 所属板块
}
```

#### 3.3.2 个性化研判生成
通过 LLM（Qwen/DeepSeek/GPT）结合用户画像生成：

**Prompt 模板：**
```
你是一位专业的 AI/科技领域投资分析师。

用户画像：
- 关注领域：{focusAreas}
- 当前持仓：{holdings}
- 投资风格：{investmentStyle}
- 风险偏好：{riskTolerance}
- 投资周期：{investmentHorizon}

最新情报摘要：
{intelSummary}

请基于以上信息生成：
1. 【情报研判】对当前情报的专业分析（200字内）
2. 【影响评估】对用户持仓的具体影响（逐只股票分析）
3. 【操作建议】针对用户风格的具体操作建议（买入/持有/减仓/观望）
4. 【风险提示】需要关注的风险点
```

**差异化示例：**
| 同一条 NVIDIA 财报利好消息 | 保守型用户 | 激进型用户 |
|--------------------------|-----------|-----------|
| 操作建议 | 持有观望，等回调再加仓 | 可适当加仓，设好止损 |
| 风险提示 | 估值已高，注意回撤风险 | 关注期权隐含波动率变化 |

### 3.4 趋势分析图表（必须项 ✅）

**要求：至少 1 个趋势分析图表**

使用前端图表库（ECharts / Chart.js）展示：

| 图表 | 类型 | 数据 |
|------|------|------|
| AI/科技板块热度趋势 | 折线图 | 近30天各子领域情报数量变化 |
| 个股关注度排行 | 柱状图 | 被提及最多的 AI/科技股 TOP10 |
| 情报来源分布 | 饼图 | 各数据源占比 |
| 市场情绪趋势 | 面积图 | 恐慌/贪婪指数变化（加分项） |

### 3.5 加分功能

#### 3.5.1 多数据源交叉验证
- 同一事件被 ≥2 个独立数据源报道时，标记为"已验证" ✅
- 仅单一来源时标记为"待验证" ⚠️
- 交叉验证结果影响置信度评分

#### 3.5.2 市场情绪面板（恐慌/贪婪指数）
- 综合以下维度计算 AI/科技板块情绪指数（0-100）：
  - 社交媒体情绪分析（LLM 情感分析）
  - 新闻情绪倾向（正面/负面/中性占比）
  - 市场波动率（VIX 相关）
  - 成交量异动
- 展示为仪表盘样式：极度恐慌(0-25) / 恐慌(25-45) / 中性(45-55) / 贪婪(55-75) / 极度贪婪(75-100)

#### 3.5.3 历史回测验证
- 选取历史情报 + 当时生成的操作建议
- 对比建议发出后实际股价走势
- 计算建议准确率，增强产品可信度

### 5.3 核心 API 设计

| 方法 | 路径 | 说明 |

### 5.4 LLM 集成方案

```
用户请求研判
    ↓
后端组装 Prompt（用户画像 + 情报摘要）
    ↓
调用 LLM API（Qwen/DeepSeek，HTTP POST）
    ↓
解析返回的结构化研判结果
    ↓
存储 + 返回前端展示
```

- 使用 RestTemplate 调用 LLM 的 OpenAI 兼容接口
- Prompt 中注入用户画像实现个性化
- 返回结果要求 JSON 格式，便于前端结构化展示

## 十一、风险与应对

| 风险 | 影响 | 应对 |

| 外部 API 限流/不可用 | 数据采集中断 | 准备 Mock 数据兜底，确保演示不受影响 |

| 3/29 核心链路故障 | 评委无法体验 | 部署后充分测试，保持 On Call |
| 评委提边界场景 | 现场演示翻车 | 提前准备 5-10 个边界场景的应对方案 |
